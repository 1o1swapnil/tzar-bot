# Incident Response & Disclosure Workflow

Turns engagement evidence into a structured disclosure packet designed to support
external vulnerability-reporting mandates — notably CERT-In's **6-hour** reporting
constraint (Section 70B, Directions 28 Apr 2022). Complements `dfir` (forensic
acquisition / IOC primitives) and `mid-engagement-ir-detection`; this skill is the
**packet-assembly + deadline** layer that strings those into a deliverable.

Closes gap #4 in `docs/compliance/CERT-In-CIGU-2026-0003-mapping.md`.

## Conventions

- `OUTPUT_DIR` from `init-engagement.py`. Engagement logs are expected in
  `$OUTPUT_DIR/logs/`.

```bash
mkdir -p "$OUTPUT_DIR/artifacts/ir"
```

## Workflows

### 1. Extract IOCs (Indicators of Compromise)

Pull IPs, domains, and hashes from the engagement logs to seed the containment packet.

```bash
mkdir -p "$OUTPUT_DIR/artifacts/ir"

# IPv4 addresses
grep -E -h -o '([0-9]{1,3}\.){3}[0-9]{1,3}' "$OUTPUT_DIR/logs/"*.log 2>/dev/null \
  | sort -u > "$OUTPUT_DIR/artifacts/ir/iocs_ips.txt"

# Domains
grep -E -h -o '([a-zA-Z0-9-]+\.)+[a-zA-Z]{2,}' "$OUTPUT_DIR/logs/"*.log 2>/dev/null \
  | sort -u > "$OUTPUT_DIR/artifacts/ir/iocs_domains.txt"

# SHA-256 hashes
grep -E -h -o '\b[a-fA-F0-9]{64}\b' "$OUTPUT_DIR/logs/"*.log 2>/dev/null \
  | sort -u > "$OUTPUT_DIR/artifacts/ir/iocs_hashes.txt"

wc -l "$OUTPUT_DIR/artifacts/ir/"iocs_*.txt
```

### 2. Generate Incident / Zero-Day Disclosure Packet

Produces a structured report: interim safeguards, mitigation, detection guidance, IOCs.

```bash
mkdir -p "$OUTPUT_DIR/artifacts/ir"
cat > "$OUTPUT_DIR/artifacts/ir/CERT_IN_REPORT_DRAFT.md" <<'EOF'
# Incident Disclosure Packet

**Incident Timestamp:** [FILL]
**Severity:** CRITICAL
**Incident Type:** Active Exploitation / Zero-Day

## 1. IOCs (Indicators of Compromise)
- Attached: `iocs_ips.txt`, `iocs_domains.txt`, `iocs_hashes.txt`

## 2. Temporary Mitigation & Containment
- [Action 1: WAF rule blocking the identified attack signature]
- [Action 2: Isolate the affected AI/decision subnet from internal systems]
- [Action 3: Trigger model failover / fallback to rule-based logic]

## 3. Detection Guidance
- YARA / Nuclei templates attached for internal scanning
  (generate with `tools/gen-nuclei-template.py`).

## 4. Impact Assessment
- [Affected systems, data classes, exposure window]

---
*Generated by the incident-response workflow.*
EOF
echo "[+] Draft written: $OUTPUT_DIR/artifacts/ir/CERT_IN_REPORT_DRAFT.md"
```

### 3. Reporting Countdown (6-hour CERT-In Helper)

Stamp the detection time and compute the reporting deadline, then push the alert through
the platform's sanctioned webhook fan-out (`tools/notify.py`) so a human meets the clock.

```bash
python3 -c "
import datetime
detected = datetime.datetime.now()
deadline = detected + datetime.timedelta(hours=6)
print(f'[INCIDENT] Detected at:  {detected:%Y-%m-%d %H:%M:%S}')
print(f'[MANDATE]  CERT-In 6h deadline: {deadline:%Y-%m-%d %H:%M:%S}')
print('--> Prepare and submit the disclosure packet before the deadline.')
" | tee "$OUTPUT_DIR/logs/ir_countdown_alert.log"

# Fire the alert via the platform webhook helper (Slack/Discord/Teams/HTTP):
python3 tools/notify.py --output-dir "$OUTPUT_DIR" --level critical \
  --text "CERT-In 6h reporting clock STARTED — prepare disclosure packet"
```

## Notes

- The legal act of reporting to CERT-In is the organisation's; this workflow produces the
  packet and tracks the deadline so the human submits on time.
- Pair with `cve-risk-score` (severity / KEV) and `validate-finding.py` to confirm the
  finding warrants critical-disclosure treatment before starting the clock.
